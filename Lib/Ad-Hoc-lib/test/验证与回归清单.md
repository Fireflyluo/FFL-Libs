## 场景一：全连通3节点 (1网关 + 2信标)

**拓扑描述:** 三个节点全互联，每个节点都能收到另外两个节点的帧。RSSI均为-40dBm（强信号）。

**运行结果:** T+5s内两个信标完成入网；网关在30s周期内发出449次A V=0帧，收到907次A/D帧。T+30s网关锁定。

|------|------|------|------|------|
| 1 | 网关启动后周期性发送 A V=0 | 每T5周期（60ms）发出一个A帧 | 30s内TX A=449次 (约15Hz) | ✓ |
| 2 | 信标在ST1状态下收到网关A帧 | 信标识别网关信号，进入U1状态 | ST1→U1 (3ms内触发) | ✓ |
| 3 | 信标向网关发送入网请求 (A V=1) | 每T5周期发送一个入网请求帧 | 信标TX A记录确认发送 | ✓ |
| 4 | 网关收到入网请求后回复确认 (flag=7) | 网关在下一A V=0帧中携带确认ID | 网关RX A记录确认接收 | ✓ |
| 5 | 信标收到带有 flag=7+自身ID 的确认 | 信标转入C1状态 | U1→C1 (~85ms内完成) | ✓ |
| 6 | 已确认信标持续广播 A V=1 | 每T5周期广播，携带flag=6+上级ID | 信标TX A持续记录 | ✓ |
| 7 | 组网窗口(30s)到期后网关停止广播 | gateway_network_locked=1 | T+30s时gw_locked=1 | ✓ |
| 8 | 信标在C1状态下提交业务数据 | 数据帧(D帧)通过信道到达网关 | DATA SUBMIT每秒1次 | ✓ |
| 9 | 网关收到数据后回复ACK | ACK帧(D V=0)包含确认信息 | DATA ACKED正常出现 | ✓ |

| BCN4连续提交 | 35次(有缺口) | **40次(连续无缺)** |
| 延迟随级数增长 | 152→200→421→609ms | 87→124→146→129ms(平坦) |

### 关于 mode5 RETRY_EXHAUSTED=475

修复后16节点全连通+60ms高频下出现475个RETRY_EXHAUSTED，**不是回归**。原因：

- 15信标×60ms = 250次/秒提交，网关ACK容量20次/秒
- 修复前级联转发让同级信标间互相"帮转"，原始数据监听确认提前收敛，掩盖了ACK瓶颈
- 修复后去掉同级无效转发，ACK统计真实可信

该值反映的是**系统容量极限**（网关ACK吞吐上限），而非回归错误。

**回归结论:** 转发语义修复对所有场景无引入回归。链式拓扑级联转发完全消除。
